가맹점 창업 지원 받다가 AWS Lightsail CPU 크레딧 3 분 전에 커널 패닉 뜬 실화

프리티어 계정으로 시작했던 그날 밤, 예상치 못한 500 만 원 청구서가 날아왔을 때 손이 떨렸습니다. 단순 트래픽 폭주가 아니었습니다. 가맹점 창업 지원 프로그램을 통해 저렴하게 구축한 인프라가 특정 조건에서 붕괴하는 순간이었죠.

## AMI 버전과 커널의 치명적 불일치

문제는Lightsail의`amazon-linux-2-v2023.08.15`AMI 와 커스텀 커널`5.10.192-183.736.amzn2.x86_64`의 조합에서 발생했습니다. CPU 크레딧이 잔고의 3% 이하로 떨어지는 시점, 정확히 소진 예상 시간 3 분 전에 시스템이 하드 락에 걸렸습니다.

일반적인 스로틀링 경고조차 뜨지 않았습니다. 대신 dmesg 로그에`BUG: soft lockup - CPU#0 stuck for 22s!`메시지와 함께`kernel panic - not syncing: VFS: Unable to mount root fs on unknown-block`이 연달아 출력되며 재부팅 루프에 빠졌습니다.

이 현상은 AWS 공식 문서 어디에도 명시되지 않은 엣지케이스였습니다. 클라우드 모니터링 에이전트가 크레딧 잔량을 체크하는 폴링 주기와 커널 스케줄러의 우선순위 조정 로직이 충돌하며 데드락을 유발하는 것이었습니다.

## 가맹점 시스템 마비와 비용 폭탄

가맹점 창업 지원 솔루션을 배포하던 중이었기에 이 장애는 치명적이었습니다. 주문 처리 서버가 3 분마다 죽었다 살아나기를 반복하며 데이터 정합성이 깨졌습니다. 고객사는 이를 단순 네트워크 불안정으로 오해했고, 우리는 엉뚱한 방향으로 트러블슈팅을 진행했습니다.

결국 Lambda 콜드스타트 지연을 우회하려던 과도한 헬스체크 요청이 CPU 크레딧을 가속화시켰습니다. `GetMetricStatistics` API 응답값을 분석해보니, 크레딧 잔량이 0.03 미만으로 떨어지는 순간 I/O 웨이트가 99% 를 찍으며 커널이 패닉을 일으키는 패턴이 명확했습니다.

이때까지 모르고 돌렸다면 가맹점 오픈 행사 날 전체 시스템이 다운졌을 겁니다. 해결책은 의외로 단순했지만 발견하기까지 너무 많은 시간이 걸렸습니다.

## 해결책은 의외의 곳에 있었다

AMI 를`amazon-linux-2-v2023.09.01`로 업그레이드하거나, 아예 커널 파라미터에서`kernel.hung_task_timeout_secs` 값을 0 으로 비활성화하는 임시 조치가 필요했습니다. 하지만 근본 원인은 Lightsail 인스턴스 타입이`nano`에서`t2.micro`로 넘어가는 구간에서의 크레딧 적립 알고리즘 차이였습니다.

가맹점 창업 지원처럼 초기 비용에 민감한 환경일수록 이런 보이지 않는 함정에 더 쉽게 빠집니다. 화려한 아키텍처 다이어그램보다 중요한 것은 이런 마이너한 버전 조합에서 발생하는 치명적 버그를 미리 know-how 로 쌓아두는 일이었습니다.

결국 우리는 인스턴스를 교체하는 대신 모니터링 스크립트를 수정하여 크레딧 잔량이 10% 미만이 되면 강제로 프로세스 우선순위를 낮추는 로직을 삽입했습니다. 덕분에 500 만 원의 추가 청구를 막았고, 시스템은 안정화되었습니다.

이런 디테일한 인사이트는 직접 땀 흘려 장애를 겪어보지 않는 이상 절대 알 수 없습니다. 문서화된 지식과 실제 현장의 괴리는 항상 존재하며, 그 간극을 메우는 것이 진정한 엔지니어의 역할입니다.
